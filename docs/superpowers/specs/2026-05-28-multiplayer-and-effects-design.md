## 批次 1 — 伺服器 + 房間系統

### 技術選型
- **後端**：Node.js + Socket.io
- **部署**：Render.com 免費方案
- **注意**：Render 免費方案閒置 30 分鐘後進入休眠，第一位玩家進入時需等待約 30 秒喚醒

### 新增檔案

| 檔案 | 說明 |
|------|------|
| `server.js` | Socket.io 伺服器主程式 |
| `package.json` | Node.js 依賴與 Render 啟動設定 |

### 房間機制

**開房間：**
- 產生一個 6 碼英數字房號（例如 `BBQ42K`），房號本身即為通行碼
- 主機看到房號後分享給朋友

**加入房間：**
- 輸入 6 碼房號 + 暱稱即可加入
- 若房號不存在或房間已開始則顯示錯誤

**房間狀態機：**
```
waiting（等待玩家） → playing（遊戲中） → ended（結束）
```

### 同步模型（Client-Authoritative）

- 每個客戶端自己控制自己的角色（本地物理不變）
- 以 **20Hz** 向伺服器發送自己的狀態快照
- 伺服器將所有玩家狀態廣播給房間內所有人
- 每個客戶端根據收到的資料渲染其他玩家
- **AI 由主機（host）負責執行**，以相同格式送出 AI 狀態，對其他客戶端與真實玩家無異

### 同步的玩家狀態欄位

```js
{
  id,          // socket.id 或 AI 識別碼
  x, y,        // 位置
  alive,       // 是否存活
  hp,          // 血量
  team,        // 'red' | 'blue' | null（個人制）
  charKey,     // 角色 key（'bacon', 'beef'...）
  isAI,        // boolean
  effects: {   // 當前 active effects
    foil, stun, slow, toxic, oilSlick
  },
  ultActive,   // 是否大絕中
  facing: { x, y }  // 面向方向
}
```

### 伺服器事件表

| 事件（client → server） | 說明 |
|--------------------------|------|
| `create_room` | 建立房間，回傳房號 |
| `join_room` | 加入房間（帶房號 + 暱稱） |
| `player_update` | 送出自己的狀態快照（20Hz）|
| `room_settings` | 主機更新房間設定 |
| `start_game` | 主機發出開始指令 |
| `game_event` | 淘汰、道具拾取等一次性事件 |

| 事件（server → client） | 說明 |
|--------------------------|------|
| `room_joined` | 確認加入，回傳房間完整狀態 |
| `player_list` | 當前玩家列表更新 |
| `game_state` | 所有玩家狀態廣播（20Hz）|
| `game_start` | 遊戲開始 |
| `game_over` | 遊戲結束 + 勝負結果 |
| `player_disconnected` | 有人斷線 |

## 批次 2 — 大廳 UI + 陣營選角 + 房間設定

### 畫面流程

```
開啟網頁（index.html）
  └─ 大廳畫面（Lobby Screen，覆蓋在 Phaser canvas 之前）
       ├─ [開房間] → 輸入暱稱 → 等待室（顯示房號）
       └─ [加入房間] → 輸入房號 + 暱稱 → 等待室

等待室（Waiting Room）
  ├─ 頂部：房號顯示（可點擊複製）
  ├─ 左側：玩家列表
  │    └─ 每個玩家：頭像 + 暱稱 + 隊伍標籤 + AI/真人標記
  ├─ 中間：角色選擇格（10 個角色，可重複選）
  ├─ 右側（僅主機可操作）：房間設定
  │    ├─ 遊戲人數（1–8，AI 自動補足空位）
  │    ├─ 遊戲模式：個人制 / 隊伍制
  │    ├─ 大絕招：一開始就可用 ✓ / 需收集能量球
  │    └─ [開始遊戲] 按鈕（所有人準備後才亮起）
  └─ 隊伍制時：玩家點自己名字旁的紅/藍圖示切換陣營
```

### 角色選擇規格

- 顯示所有 10 個角色圖示
- 點擊即選取，可重複選（多人選同角色允許）
- 選取後顯示隊伍顏色外框（紅框 / 藍框 / 無框）

### 房間設定欄位

```js
roomSettings = {
  maxPlayers: 8,          // 1–8，AI 補足
  gameMode: 'individual', // 'individual' | 'team'
  ultFromStart: false,    // true = 大絕一開始即可用
  teamCount: 2,           // 固定 2 隊（紅、藍）
}
```

### AI 補位規則

- `maxPlayers - 真人數量 = AI 插槽數`
- AI 隊伍由主機在等待室拖拉分配（隊伍制）
- AI 角色隨機選取（不與主機選的重複，若全部重複則允許重複）

### 遊戲模式勝負規則

**個人制：**
- 最後一位存活玩家獲勝
- 與現有邏輯相同

**隊伍制：**
- 同隊可以互相傷害（誤傷保留，增加趣味）
- 一支隊伍全員被淘汰，另一隊勝利
- 若最後剩兩隊各一人，繼續到一方淘汰

### 斷線處理

- 真人斷線：該玩家位置凍結 3 秒，之後以 AI 接管（繼承當前血量與位置）
- 主機斷線：伺服器指定第一個加入的真人玩家成為新主機（接管 AI 控制）

## 批次 3 — 地圖效果升級

### 效果 1：油脂噴濺強化

**觸發時機：** 任何推擠碰撞發生時（現有 push 邏輯）

**新增效果：**
- 碰撞點生成 **4–6 顆油滴粒子**
- 每顆油滴以隨機方向（±60° 範圍內）飛出，飛行距離 30–60px
- 落地後生成一個小 `oilSlick`：半徑 12px，持續 2.0 秒，顏色 `#ffaa00`（飽和橘黃）
- 玩家踩到小 oilSlick 輕微減速（`slowMul: 0.85`，0.5 秒）

**與現有 oilSlick 系統的差異：**
- 現有：五花肉大絕的大油漬（半徑較大，持續較長）
- 新增：推擠產生的小油漬（半徑 12px，較短暫）
- 兩者共用 `oilSlicks[]` 陣列，以 `type: 'small' | 'large'` 區分

### 效果 2：群聚火焰

**觸發條件：** 某區域內（半徑 80px）累積的小 oilSlick 數量 ≥ 5 個

**實作方式：**
- 每幀掃描 `oilSlicks[]`，對每個 oilSlick 計算其 80px 範圍內的同類數量
- 達到門檻時，在群聚中心點生成 `heatZone`

**heatZone 規格：**
```js
{
  x, y,
  radius: 55,          // 視覺半徑
  life: 動態，         // 隨 oilSlick 消耗而縮減
  intensity: 1–3,      // 根據油漬數量（5=1, 8=2, 12+=3）
}
```

**視覺：**
- 橘紅色火焰粒子從地板向上竄（多層半透明圓，每 0.08 秒更新位置）
- `intensity` 越高，粒子越多、顏色越白熱
- 火焰繪製於玩家層之下（`depth: 5`）

**效果：**
- 進入 heatZone 的玩家每秒扣 `2 HP`（少量，製造壓力但不致命）
- heatZone 存在期間，其範圍內的 oilSlick 加速消耗（`life` 扣除速度 ×2）
- 所有 oilSlick 消耗完後 heatZone 消失

## 批次 4 — 工作日誌

### 架構

**`CHANGELOG.md`（手動維護）：**
- 每次修改遊戲後，在此檔案頂部新增一筆紀錄
- 格式：`## YYYY-MM-DD` + 條列式更動內容

**`changelog.html`（自動渲染）：**
- 放在 GitHub Pages 根目錄
- 用 `fetch('CHANGELOG.md')` 抓取內容，用 `marked.js` 渲染成 HTML
- 樣式：時間軸風格，每次更新一個卡片

**入口：**
- 路徑 A：直接訪問 `https://kshwin.github.io/bbq-survival-game/changelog.html`
- 路徑 B：遊戲主畫面右上角加「📋 更新日誌」按鈕，點擊後開新分頁
